# Deploying ClinicFlow

Recommended first deployment: **Render** (managed Postgres + Docker + free TLS,
least setup). The repo includes `render.yaml` that provisions everything.

## A. One-click-ish via Blueprint (uses render.yaml)

1. Push the repo to GitHub (already connected).
2. In Render → **New → Blueprint** → pick this repo. Render reads `render.yaml`
   and shows: a Postgres DB, the API (Docker), and the frontend (static).
3. Click **Apply**. Render will:
   - create the `clinicflow-db` Postgres,
   - build the backend image from `Dockerfile` and inject the DB connection,
   - build the frontend and serve `frontend/dist`.
4. First boot: Flyway creates the `global` schema + tables; the tenant migration
   runner runs (no clinics yet, so nothing to migrate).

### URLs
- API:      `https://clinicflow-api.onrender.com` (Swagger at `/swagger-ui.html`)
- Frontend: `https://clinicflow-frontend.onrender.com`

> If you rename the services, update **two** values: `APP_CORS_ORIGINS` on the
> API and `VITE_API_BASE_URL` on the frontend, so they point at each other.

## B. Set the integration secrets

In the Render dashboard → `clinicflow-api` → **Environment**, fill the
`sync:false` keys when you have them (the app runs without them, in fallback mode):

- `RAZORPAY_KEY_ID`, `RAZORPAY_KEY_SECRET`, `RAZORPAY_WEBHOOK_SECRET`, `RAZORPAY_PLAN_*`
- `MSG91_AUTH_KEY`, `MSG91_TEMPLATE_ID`   (real OTP SMS)
- `GUPSHUP_API_KEY`, `GUPSHUP_SOURCE`     (WhatsApp)

Then point your Razorpay webhook at:
`https://clinicflow-api.onrender.com/api/public/webhooks/razorpay`

## C. Verify

1. Open the frontend URL → **Register a clinic** (OTP shows in dev only; in prod
   you need MSG91 configured, or temporarily set `DEV_TOOLS=true` to test).
2. Log in with phone + password → run a patient → appointment → bill.
3. Check `https://clinicflow-api.onrender.com/actuator/health` → `{"status":"UP"}`.

## Notes & production hardening
- **Free tier** sleeps when idle and the free Postgres expires after 90 days —
  fine for a pilot; upgrade both to paid for a real clinic.
- Keep `DEV_TOOLS=false` in production (it would expose OTPs and the subscription
  simulate controls otherwise).
- `JWT_SECRET` is auto-generated by Render; never commit secrets.
- For multi-instance scaling later, set `OTP_STORE=redis` + add a Render Redis and
  `REDIS_HOST`/`REDIS_PORT`.

## Alternative hosts
- **Railway** — similar: add a Postgres plugin, deploy the Dockerfile, set the
  same env vars; deploy the frontend as a static site.
- **Fly.io** — `fly launch` on the Dockerfile, attach Fly Postgres (`fly pg create`),
  set env via `fly secrets set`. Frontend can go to any static host (Netlify/Vercel/Cloudflare Pages)
  with `VITE_API_BASE_URL` set to the API URL.
- **Any VPS** — `docker compose up -d` works as-is; put nginx/Caddy in front for TLS.
