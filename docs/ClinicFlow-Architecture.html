1 · Overview & Goal

ClinicFlow lets small and medium Indian clinics run their front desk and consultations digitally. A single deployment serves many clinics ("tenants"), each fully isolated in a dedicated PostgreSQL schema. The product is deliberately India-specific: phone-OTP login, a token-based OPD queue, GST/HSN invoicing, ABHA health IDs, NMC doctor registration, WhatsApp follow-up reminders, and Razorpay subscriptions.

What a clinic gets

  • Self-service signup with a 90-day trial
  • OTP login for every staff member
  • Patient registry with auto UHID
  • Daily token queue per doctor
  • Vitals, prescriptions, GST bills
  • Automated follow-up reminders

Why schema-per-tenant

  • Hard isolation — one clinic can never read another's rows
  • Simple queries — entities carry no tenant_id; routing is at the connection layer
  • Per-tenant ops — backup, restore, or migrate one clinic independently
  • Cross-tenant jobs — loop over schemas for system-wide tasks (reminders)

3 · High-Level Architecture

Classic layered Spring Boot: controllers expose REST, services hold business logic and transactions, repositories are Spring Data JPA interfaces. The twist is the multi-tenancy plumbing between Hibernate and PostgreSQL that swaps the active schema per request.

Request flows top-down through the filter chain, controllers, services, and repositories; Hibernate's tenancy layer rewrites the active schema before SQL hits Postgres.

4 · Multi-Tenancy Deep Dive

Three small pieces cooperate to make every JPA query land in the right clinic's schema, with entities staying blissfully unaware of tenancy.

ComponentResponsibility
TenantContextA ThreadLocal<String> holding the current request's schema name. Set by the JWT filter, cleared in a finally block to prevent thread-pool leakage.
TenantSchemaResolverImplements Hibernate's CurrentTenantIdentifierResolver. Returns TenantContext.get() or defaults to global.
TenantConnectionProviderImplements Hibernate's MultiTenantConnectionProvider. On checkout runs SET search_path TO <schema>, public; on release resets to global, public before returning the connection to Hikari.
Key design choice: entities for clinic data declare no schema, so Hibernate emits unqualified table names that resolve via search_path. Global entities (Tenant, StaffDirectory) are explicitly @Table(schema="global"), so they always hit the shared schema regardless of the active tenant.
Two settings that must stay off: hibernate.default_schema (would force every query onto one schema, defeating routing) and ddl-auto validation/update (runs against a single schema at boot). Schema management is owned entirely by Flyway. open-in-view is also disabled so sessions open inside the transactional method, after the tenant is set.

5 · Request Lifecycle (tenant routing)

JWT → ThreadLocal → Hibernate resolver → SET search_path → SQL runs in the clinic schema → context cleared on the way out.

Solid lines = foreign keys. medicine_templates and consultations tables also exist in the schema (seeded medicines; consultation notes) and are not yet mapped to JPA entities.
Concurrency-safe IDs: UHIDs (CF-00001) and invoice numbers (INV-2026-00001) come from per-schema Postgres sequences (patient_uhid_seq, bill_invoice_seq), not COUNT(*), so simultaneous registrations can't collide. A unique index on (visit_date, doctor_id, token_number) guards daily tokens.

7 · Auth & OTP Flow

Authentication is phone-OTP. The returned JWT carries the staff id (subject), role, and tenantId (the clinic schema). That token is the only thing the rest of the API needs to route to the right clinic.

OTPs are single-use and expire in 10 minutes. The clinic is resolved via the global staff_directory (fallback: tenants.owner_phone), so any staff member — not just the owner — can log in.

8 · Clinic Provisioning Flow

Signup is self-service and creates a brand-new isolated schema on the fly.

  1. POST /api/public/clinics/register with clinic + owner details.
  2. TenantProvisioningService generates a schema name (tenant_<phone>) and checks it's free.
  3. CREATE SCHEMA for the clinic.
  4. Flyway runs the tenant migrations (V1 tables, V2 sequences/constraints) against that schema.
  5. The clinic row is saved in global.tenants.
  6. The owner Staff row is created in the new schema, and the owner phone is added to global.staff_directory.
  7. Owner logs in via OTP → gets a JWT scoped to the new schema.
Atomicity caveat: schema DDL can't roll back inside the JPA transaction, so a failure after step 3 can leave a half-provisioned clinic (schema + tenant row, no staff). Hardening this into an all-or-nothing flow with cleanup-on-failure is a recommended follow-up.

9 · Appointment Booking Flow

  1. Receptionist registers/looks up the patient (POST /api/patients) → UHID assigned from the sequence.
  2. POST /api/appointments with patientId + doctorId + opFee.
  3. Service assigns the next token for that doctor/day and saves the appointment (status WAITING).
  4. If opFee > 0, an OP-fee Bill (+ BillItem) is created in the same transaction, invoice number from the sequence.
  5. Doctor/reception read the queue (GET /api/appointments/today) and advance status to CONSULTING / COMPLETED.
Authorization split: creating appointments is restricted to RECEPTIONIST / ADMIN (front-desk intake), while doctors and nurses can read queues and update status. An ADMIN passes every check.

10 · Security & Roles

Stateless JWT. JwtAuthFilter validates the token, sets the tenant, and builds the Spring Security principal with authority ROLE_<role>. Method-level @PreAuthorize enforces per-endpoint roles.

RoleCan do
ADMINEverything — staff management, intake, queues, status
RECEPTIONISTRegister patients, book appointments, view/update queue
DOCTORView patients & queues, update appointment status
NURSEView patients & queues (read-oriented)
Before production: move the JWT secret and DB password fully to environment/secret storage (defaults exist for dev), put OTPs in Redis, and add rate-limiting on /api/auth. The HS256 secret must be ≥32 bytes.

12 · Swagger UI

Interactive docs are generated from the controllers and served by the running app:

ResourceURL
Swagger UIhttp://localhost:8080/swagger-ui.html
OpenAPI JSONhttp://localhost:8080/v3/api-docs

Using it

  1. Open Swagger UI. The Auth and Clinic Onboarding groups are public — call verify-otp there and copy the token.
  2. Click Authorize (top-right), paste the token, confirm. Every secured call now sends Authorization: Bearer <token>.
  3. The token's tenantId automatically scopes all calls to your clinic — no extra parameters needed.
Because the spec is generated from the actual controllers and @Operation annotations, the docs never drift from the code.
